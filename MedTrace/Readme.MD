# MedTrace

**A full-stack medication and symptom tracker with caregiver support and an AI summarization layer.**

MedTrace helps patients log medications and daily symptoms, surfaces real drug information, and generates plain-English reports they can bring to their doctor. A caregiver mode lets a family member, with the patient's permission, monitor a patient's data and enter information on their behalf. The app is designed to *organize and communicate* health information. It never diagnoses or recommends medical decisions.

**Live app:** [medtrace.example.com](https://medtrace.example.com) &nbsp;|&nbsp; **Backend API:** [api.medtrace.example.com](https://api.medtrace.example.com)

---

## Why this project

Medication non-adherence is a large, under-addressed public health problem, and patients often arrive at appointments without a clear record of how they have been feeling. MedTrace closes that gap with a simple logging flow and AI-generated summaries, while deliberately keeping the AI in a *summarizing*, not *advising*, role. The result is a complete product: a patient app, a caregiver app, a hardened Java backend, an external drug-data integration, an AI layer with auditable guardrails, and PDF reporting, all deployed and running.

## Highlights for a quick read

- **Full-stack and deployed.** React Native + Expo client (mobile and web) talking to a Java 25 / Spring Boot backend on PostgreSQL, live in production.
- **Real external data.** Adding a medication pulls live purpose, warnings, and side effects from the OpenFDA drug label API.
- **An AI layer that stays honest.** Trend detection runs as transparent rules in code; the model only phrases findings the code already produced. It never decides what is medically notable, never diagnoses, and never advises.
- **Retrieval-augmented summaries.** Drug references are grounded in the patient's own FDA facts using semantic vector search (pgvector + Gemini embeddings) rather than the model's training data.
- **A real authorization model.** Patient-initiated caregiver links with two permission levels, enforced by central view and input gates, with full audit attribution on every caregiver entry.
- **Reporting and alerts.** Downloadable weekly, monthly, and yearly PDF reports, plus automated caregiver email alerts via Resend.

## What the app does

- **Accounts and security.** Registration with validated input and BCrypt-hashed passwords, login issuing signed JWT access tokens, stateless JWT authentication enforced on every protected request by a custom filter, and role-based access distinguishing patient and caregiver accounts.
- **Medication management.** Full create, list, edit, and delete, scoped to the authenticated user with ownership checks. Adding a medication automatically pulls purpose, warnings, and side effects from the OpenFDA drug label API, and facts re-fetch when a medication's name changes. Every add, dosage change, switch, and removal is recorded with a timestamp, forming an audit trail.
- **Daily check-ins.** Patients log a structured snapshot (mood, energy, pain level, sleep hours, medications-taken) plus an optional free-text journal entry, capped at one check-in per day per user and viewable as dated history.
- **Schedules and dose logging.** Patients define which times of day a medication should be taken, then log individual doses as they take them. As-needed (PRN) medications are supported with no fixed schedule.
- **Dose status detection.** For any day, the app reports each scheduled dose as taken, upcoming, overdue, or missed, computed against the schedule and a grace period, so a patient or caregiver can see adherence at a glance. An overdue dose is one that is late but still actionable today; a missed dose is one a past day closed out without a log.
- **AI health narratives.** An on-demand, plain-English summary of a patient's recent check-ins, generated by Google Gemini under strict guardrails: describe only, never diagnose or advise.
- **AI trend detection.** Transparent, code-owned rules flag notable patterns such as rising pain, sustained low mood, and missed-dose clusters. Gemini only phrases the already-detected findings, so the medical judgment stays in auditable code rather than the model.
- **Retrieval-grounded narratives.** An alternate narrative that grounds the AI summary in the patient's own FDA drug-label facts, retrieved by semantic vector search (pgvector + Gemini embeddings), so drug references draw on authoritative data rather than the model's training.
- **PDF health reports.** On-demand weekly, monthly, and yearly reports a patient can download and bring to an appointment, covering current medications, per-medication adherence rates, averaged check-in trends, and a clearly fenced AI summary of the patient's own journal entries that degrades gracefully if the AI is unavailable. Rendered from HTML and CSS via openhtmltopdf.
- **Caregiver mode (patient-initiated).** A patient can invite a caregiver to their account at one of two permission levels (`VIEW_ONLY` or `VIEW_AND_INPUT`). The caregiver accepts, and either party can revoke. With `VIEW_AND_INPUT`, the caregiver can log check-ins and doses on the patient's behalf, every entry stamped with who entered it. With either permission, the caregiver can view the patient's check-ins, dose history, dose status, and medications. All caregiver access is enforced by a central authorization gate.
- **Caregiver email alerts.** When the app detects a cluster of missed doses for a patient, linked caregivers receive an automated email alert via Resend, reusing the same transparent trend-detection rules that power the in-app flags.
- **Consistent error handling.** A global exception handler returns clean, readable JSON error responses with appropriate HTTP status codes.

## Architecture

The backend follows a standard layered Spring Boot structure:

```
controller  ->  service  ->  repository  ->  database
   (HTTP)       (logic)      (data access)   (Postgres)
```

**Authentication flow.** A request arrives, the JWT filter checks the `Authorization` header, and if a valid token is present it loads the user and marks the request authenticated. Spring Security then allows or denies based on the route's rules and the user's role.

**Drug enrichment flow.** When a medication is added (or its name changed), the service calls the OpenFDA label API, parses the response, and stores the relevant fields alongside the medication. If a drug is not recognized, the medication still saves and drug facts simply remain empty (graceful degradation).

**Dose status flow.** Dose status is computed on demand rather than stored. For a requested date, the service loads the user's active schedules and the doses logged that day, then classifies each scheduled time. A logged dose is taken regardless of the clock, which lets late logging self-correct. An unlogged dose is upcoming before its time, overdue once its time plus a grace period has passed on the current day, and missed once the day itself has closed. Because the status is derived at read time, it always reflects the latest logs without a background job, and an overdue dose never counts against the adherence rate until the day actually ends.

**AI layer flow.** The narrative and trend features gather a recent window of the patient's own data, then call Google Gemini through a thin integration service. Two design choices define this layer. First, separation of plumbing from logic: a `GeminiService` handles only the API call (auth via header, raw-string response parsed manually, graceful handling of the model's transient busy responses), while feature services decide what to ask. Second, and most importantly, the model never decides what is medically notable. Trend detection runs deterministic, transparent rules in application code (for example, pain rising past a threshold, or a cluster of missed doses), and Gemini is used only to phrase the already-detected findings warmly. The system prompts forbid diagnosis, advice, and causal claims, and every output ends with a not-medical-advice reminder.

**Retrieval-augmented flow (grounded narratives).** Drug-fact text is converted to embedding vectors via Gemini's embedding model and stored in a pgvector `vector` column. At narrative time, a query built from the patient's medication names is embedded and used to retrieve the most semantically similar drug-fact chunks (cosine distance via pgvector's `<=>` operator), scoped to the patient's own medications. The retrieved facts are injected into the prompt as factual grounding. Two correctness details matter: embeddings are L2-normalized in application code (required at non-default vector sizes, or cosine similarity is distorted), and the grounded system prompt explicitly permits citing drug facts as neutral background while still forbidding turning them into advice, a boundary verified to hold even when a relevant drug fact sits next to a matching reported symptom.

**Reporting flow (PDF).** A single report service gathers a period's worth of data and assembles it into an HTML document. A separate, deliberately tiny renderer converts that HTML to PDF bytes, so the PDF library is swappable in one place, and the controller exposes one endpoint that takes a period parameter and returns the finished PDF with download headers. The period (weekly, monthly, yearly) resolves to a start date and reuses the existing data queries as a date filter, so no new persistence was needed. Two correctness choices mirror the rest of the app's honesty instinct: not-yet-due doses are excluded from the adherence rate rather than counted as missed, and each averaged check-in metric is computed only over the days that field was actually recorded, with the contributing day-count shown alongside it. The AI journal summary is fenced into its own bordered section with a non-clinical disclaimer in the section header, uses the plain (non-grounded) narrative, is patient-version only, and degrades to a plain unavailable message if the AI service is briefly down. The data-only sections never depend on it.

**Caregiver flow (care links and authorization gate).** A `CareLink` row connects a patient and a caregiver with a permission level (`VIEW_ONLY` or `VIEW_AND_INPUT`) and a status (`PENDING`, `ACTIVE`, or `REVOKED`). The relationship is patient-initiated: the patient creates the invite (PENDING), the caregiver accepts (ACTIVE) or declines (REVOKED), and either party can revoke. A revoked link can be re-invited, reusing the same row to respect a one-link-per-pair unique constraint, which restarts the consent handshake rather than silently restoring access. All caregiver-on-behalf access funnels through two central gate methods on the care-link service: `resolveInputTarget` (caller is the patient, or holds an ACTIVE `VIEW_AND_INPUT` link) for writes, and `resolveViewTarget` (caller is the patient, or holds any ACTIVE link) for reads. The single difference between them, whether `VIEW_AND_INPUT` is required, is the entire meaning of the two-level permission model, kept legible by the two methods reading near-identically. Caregiver-entered check-ins and doses carry an `enteredBy` field (null when the patient entered it themselves), so a caregiver's entry is never silently attributed to the patient and is visible as such when the data is read back.

**Alerting flow (caregiver email).** The same transparent missed-dose detection that powers the in-app trend flags drives caregiver notifications. When a patient's missed-dose count crosses the threshold, the app composes a plain-language alert and sends it to linked caregivers through Resend. The alert describes the pattern in the patient's own data and never offers medical advice.

**Data integrity.** The daily check-in feature enforces one entry per user per day at two levels: a database unique constraint as a hard backstop, and a service-layer check that returns a friendly error before the constraint is hit. Dose logging applies the same defensive pattern: a scheduled dose can be logged only once per day, while as-needed doses are deliberately exempt. Ownership is verified at every step. A dose can only be logged against a medication the user owns, and only against a schedule that belongs to that same medication. When a caregiver logs on a patient's behalf, the medication-ownership check runs against the resolved patient, not the caregiver. Retrieval is likewise owner-scoped: a patient's grounding context only ever draws from their own medications' facts.

**Secrets handling.** The database password and external API keys (Gemini, Resend) are read from environment variables, never committed to source. Keys are configured without a fallback so the application fails fast if one is missing rather than running unauthenticated.

## Client design

The mobile and web client is one Expo codebase with two role-based experiences. Authentication context resolves who the user is and persists the session in secure storage so the app stays logged in across restarts. For caregivers, a second context resolves which linked patient is being viewed and carries that identity into every request. The patient experience centers on a daily home view (today's doses with a one-tap mark-as-taken, and a check-in prompt), with tabs for medications, check-ins, reports, and profile. The caregiver experience mirrors it for monitoring: a linked-patient overview, a read-only patient view, an on-behalf logging tab gated by permission, reports, and profile. A first-launch guided tour introduces the main features and can be replayed from the profile screen.

## Running locally

**Prerequisites:** Java 17+ (built on Java 25), Maven, Node.js with the Expo CLI, and PostgreSQL with the **pgvector** extension installed.

### Backend

1. Create the database and enable pgvector:
   ```sql
   CREATE DATABASE medtrace;
   \c medtrace
   CREATE EXTENSION vector;
   ```
2. Configure `backend/src/main/resources/application.properties`, or supply credentials via environment variables (recommended).
3. Set the required environment variables:
   - `DB_PASSWORD` your PostgreSQL password
   - `GEMINI_API_KEY` a Google Gemini API key (AI narratives, trends, and embeddings)
   - `RESEND_API_KEY` a Resend API key (caregiver email alerts)
   - `JWT_SECRET` optional; a development fallback is used otherwise
4. From the `backend/` directory, run:
   ```bash
   ./mvnw spring-boot:run
   ```
5. The API starts on `http://localhost:8080`.

### Client

1. From the `mobile/` directory, install dependencies:
   ```bash
   npm install
   ```
2. Point the client at your API by setting the base URL in `src/api/config.ts` (use your machine's LAN IP when testing on a physical device).
3. Start Expo:
   ```bash
   npx expo start
   ```
4. Open the app on a device with Expo Go, in an iOS or Android simulator, or in the browser with the web build.

## Roadmap

- **Authentication and foundation:** project setup, PostgreSQL, User entity, JWT auth, role-based access
- **Medications:** medication CRUD, OpenFDA drug-fact integration, medication-change detection
- **Daily check-ins:** symptom logging (form and free-write journal), log history, global error handling
- **Schedules and dose status:** per-medication dosing schedules, dose logging (scheduled and as-needed), taken / upcoming / overdue / missed status computed at read time
- **AI layer:** Gemini-generated narratives, code-detected and AI-phrased trend flags, and a retrieval pipeline (pgvector + Gemini embeddings) that grounds summaries in the patient's FDA drug facts, with the no-advice boundary verified under grounding
- **Reports:** weekly, monthly, and yearly PDF reports rendered from HTML and CSS via openhtmltopdf, covering medications, per-medication adherence, averaged check-in trends, and a fenced-off AI journal summary that degrades gracefully when the AI is unavailable
- **Caregiver mode:** patient-initiated care-link lifecycle (invite, accept, decline, revoke, re-invite), a two-level permission model (`VIEW_ONLY` / `VIEW_AND_INPUT`) enforced by central input and view authorization gates, caregiver input on check-ins and doses with `enteredBy` audit attribution, and caregiver viewing of check-ins, dose history, dose status, and medications
- **Caregiver alerts:** automated missed-dose email notifications to linked caregivers via Resend
- **Client and deployment:** React Native + Expo client for mobile and web, containerized backend, deployed for browser access
